# Data Dictionary

> *Schema and column reference for the gold-layer tables that feed the dashboard.*

This document describes the analytical data that powers the dashboard. The data lives in **Parquet files** generated by the gold-layer pipeline (see [`../pipeline/`](../pipeline/)) and is imported into Power BI via standard file connectors.

## Why no raw data files in this repo

The actual Parquet files are not committed to the repo for three reasons:

1. **Size.** The combined gold export runs into hundreds of MB. Git is not optimized for binary file storage at that scale.
2. **Reproducibility.** The pipeline scripts in `pipeline/` regenerate the gold tables deterministically from any source data refresh. Committing the output files alongside the pipeline that produces them would mean two sources of truth.
3. **Portfolio focus.** The interesting work is in the schema design and DAX, not in inspecting raw rows. The .pbix file embeds the data needed to render the dashboard.

What's documented below is the **shape** of the data — column names, types, grain — so a reviewer reading the model can understand what's flowing through.

## About the dataset

The dataset represents a **mid-size omnichannel retailer** operating across stores, distribution centers, and an e-commerce web channel.

| Element | Value |
|---------|-------|
| SKUs | 40 |
| Categories | 4 (ELECTRONICS, GROCERY, HOUSEHOLD, PERSONAL CARE) |
| Brands | 4 (EIRA, FJORD, NORDMARK, POLAR) |
| Locations | 5 (mix of stores, DCs, web channel) |
| Date range | December 2025 – May 2027 (~18 months) |
| Sales transactions | ~57,000 |
| Inventory snapshots | ~17,000 |
| Inventory movements | ~64,000 |

The data is **synthetic but representative** — built to mirror the structure, scale, and messiness of real retail supply chain data, including realistic patterns like demand seasonality, occasional adjustment events for data quality testing, and category-level performance variation.

## Gold-layer tables

### `gold__dim_product`

One row per SKU.

| Column | Type | Description |
|--------|------|-------------|
| ProductSKU | string | Natural key — primary identifier for the SKU |
| ProductName | string | Display name shown in dashboards |
| Category | string | One of the 4 product categories |
| Brand | string | One of the 4 house brands |
| UnitCost | decimal | Per-unit cost — used in inventory value calculations |

### `gold__dim_location`

One row per location.

| Column | Type | Description |
|--------|------|-------------|
| LocationID | string | Natural key — primary identifier for the location |
| LocationName | string | Display name |
| LocationType | string | `STORE` / `DC` / `WEB` |
| Region | string | Geographic grouping |

### `gold__fact_sales`

One row per sales transaction. ~57K rows.

| Column | Type | Description |
|--------|------|-------------|
| SaleID | string | Surrogate key — unique transaction ID |
| SaleDate | date | Transaction date — joins to `dim_date` |
| ProductSKU | string | FK → `dim_product[ProductSKU]` |
| LocationID | string | FK → `dim_location[LocationID]` |
| Quantity | integer | Units sold |
| UnitPrice | decimal | Price at point of sale |
| Revenue | decimal | Derived: `Quantity × UnitPrice` |

### `gold__fact_inventory_snapshots`

Daily inventory positions per (date, location, SKU). ~17K rows.

| Column | Type | Description |
|--------|------|-------------|
| SnapshotID | string | Surrogate key |
| SnapshotDate | date | Date of the snapshot |
| ProductSKU | string | FK → `dim_product[ProductSKU]` |
| LocationID | string | FK → `dim_location[LocationID]` |
| OnHandQty | integer | Units physically present at this location on this date |

### `gold__fact_inventory_movements`

Event-level inventory movements. ~64K rows.

| Column | Type | Description |
|--------|------|-------------|
| MovementID | string | Surrogate key |
| MovementDate | date | Date of the movement event |
| ProductSKU | string | FK → `dim_product[ProductSKU]` |
| LocationID | string | FK → `dim_location[LocationID]` |
| MovementType | string | `SALE` / `REPLENISH` / `ADJUSTMENT` |
| Quantity | integer | Units moved (always positive — direction encoded by type) |

### `gold__fact_inventory_exposure`

Derived analytical fact — one row per SKU. Pre-aggregates the analytical logic so business definitions live in the gold layer rather than being scattered across DAX.

| Column | Type | Description |
|--------|------|-------------|
| ProductSKU | string | FK → `dim_product[ProductSKU]` |
| OnHandValue | decimal | Current inventory value at unit cost |
| DaysOfSupply | decimal | Days of cover based on trailing demand |
| OverstockFlag | boolean | True if DoS > 90 days |
| LowStockFlag | boolean | True if DoS < 7 days |
| ReorderPriorityScore | decimal | Velocity-weighted urgency |
| TrappedCapital | decimal | Excess inventory value above target DoS |

## Source data lineage

The gold tables are produced from upstream **silver** tables, which in turn are cleaned versions of **bronze** tables capturing raw source data. Full pipeline architecture is in [`../docs/02-architecture.md`](../docs/02-architecture.md).

Sources include:

- Sales transaction exports (line-item level)
- Daily inventory snapshot files
- Inventory movement event streams (sales, replenishments, adjustments)
- Product master reference data
- Location master reference data

The pipeline applies five standard data quality checks at the silver layer (dimension uniqueness, key coverage, date parsing, range validation, movement type validity) before promoting to gold.
